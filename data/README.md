# qsafe Data Guide

This folder is the source of truth for the framework and project assessments.

## What To Edit

Edit these files:

| File | Purpose |
|---|---|
| `taxonomy.json` | Canonical 5-core, 30-component framework. Change this only when the framework itself changes. |
| `projects/index.json` | Project registry, display order, queued projects, and parent relationships. |
| `projects/<id>.json` | One assessed chain or product. This is the main contributor file. |
| `projects/_template.json` | Starting point for a new project assessment. |

Do not hand-edit these generated fallback files:

| File | Purpose |
|---|---|
| `taxonomy.js` | Browser fallback copy of `taxonomy.json` for opening the site from `file://`. |
| `projects/bundle.js` | Browser fallback bundle of all project assessments. Regenerated by `scripts/build_projects.py`. |

## Contribution Routes

### Update One Component

Use this when one row in an existing assessment is outdated, incorrect, or needs better sources.

1. Open `projects/<project-id>.json`.
2. Find the stable component id, such as `1.3`.
3. Edit only that component block under `assessment`.
4. Update `verdict`, `scheme`, `why`, and `sources`.
5. Open a PR, or use the component update form if you do not want to edit JSON.

### Add A Chain Or Product

Use this when qsafe is missing a full chain, L2, bridge, app, wallet system, shielded pool, or other product.

1. Copy `projects/_template.json` to `projects/<id>.json`.
2. Fill the top-level project fields.
3. Fill all 30 component blocks.
4. Add the project to `projects/index.json`.
5. Run `python3 scripts/build_projects.py` if working locally.
6. Open a PR.

## Project File Shape

Each assessed project has:

```json
{
  "id": "project-id",
  "name": "Display Name",
  "type": "Short project type",
  "parent": "host-chain-id",
  "reviewed": "YYYY-MM-DD",
  "summary": "Plain-language summary",
  "links": { "Website": "https://..." },
  "assessment": {
    "1.1": {
      "verdict": "pass | fail | na",
      "scheme": "Concrete crypto or mechanism",
      "why": "Short source-backed reason",
      "sources": ["https://..."]
    }
  }
}
```

`parent` is only needed for products built on another chain. For base chains, omit it or set it to `null`.

## Verdicts

| Verdict | Meaning |
|---|---|
| `pass` | The live default setup is quantum-safe for this component. |
| `fail` | The live default setup relies on quantum-breakable crypto for this component. |
| `na` | The component is outside this project design and is excluded from the score. |

Score = `pass / (pass + fail)`. `na` components are excluded because the project does not use that surface.

## Schemas

Machine-readable schemas live in `schema/`:

| File | Purpose |
|---|---|
| `schema/project.schema.json` | Shape of `projects/<id>.json`. |
| `schema/project-index.schema.json` | Shape of `projects/index.json`. |
| `schema/taxonomy.schema.json` | Shape of `taxonomy.json`. |

The repo keeps validation zero-dependency, so `scripts/build_projects.py` enforces the important rules directly. The schema files are there for editors, agents, and external validators.

## Validation

From the repo root:

```sh
python3 scripts/build_projects.py
python3 scripts/build_projects.py --check
```

The script validates all assessed projects and regenerates `projects/bundle.js` unless `--check` is used.
